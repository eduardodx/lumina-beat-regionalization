# ClinVar Dataset Curation

This page documents the ClinVar dataset inputs, filtering rules, and metadata that are currently
encoded in the repository. It is meant to be the consolidation point for any exploratory notebooks
or ad hoc audits around ClinVar curation.

## Current Artifacts

- Source parquet: `data/datasets/clinvar/processed/clinvar_dataset.parquet`
- Window cache: generated by `eval.clinvar.dataset.build_variant_cache`
- Downstream dataset wrapper: `eval.clinvar.dataset.ClinVarFineTuneDataset`
- Notebook inventory in this repository: `notebooks/data.ipynb` only

I did not find a committed ClinVar-specific notebook bundle in the current tree. If the filter and
metadata notebooks live elsewhere, they should be copied under `notebooks/clinvar/` or summarized
here so the dataset story stays in one place.

## Source Columns

The current ClinVar parquet is expected to provide at least:

- `Chromosome`
- `Start`
- `ReferenceAlleleVCF`
- `AlternateAlleleVCF`
- `label`
- `split_within_gene`

The dataset loader also looks for optional metadata columns when present:

- `consequence_bucket`
- `ConsequenceBucket`
- `Consequence`
- `consequence`
- `most_severe_consequence`
- `GeneSymbol`
- `gene_symbol`
- `Gene`
- `gene`
- `SYMBOL`

## What Gets Filtered

The current pipeline does not apply a large semantic filter stage inside the evaluation runner.
Instead, it performs a small set of mechanical checks while building the cache:

1. Variants are split by `split_within_gene` into train and test.
2. Variant windows are extracted from hg38 using the provided allele strings.
3. Variants that do not match the FASTA reference are skipped.
4. Variants that cannot produce a full context window are skipped.
5. Regime B additionally requires PhyloP and GTF inputs to build biological features.

That means the main curation logic lives in the source parquet, not in the runner.

## Metadata Retained In Cache

The cached records currently preserve:

- `ref_seq`
- `alt_seq`
- `variant_offset`
- `label`
- `original_index`
- `split_within_gene`
- `consequence_bucket`
- `gene_symbol`
- `bio_features` for Regime B

`consequence_bucket` is taken from the source parquet when available; otherwise it is inferred from
allele lengths as `snv`, `mnv`, or `indel`.

## Notebook Structure

If we want the exploratory work to stay auditable, the notebook layout should be simple:

- `notebooks/clinvar/01_filters.ipynb` for inclusion/exclusion decisions
- `notebooks/clinvar/02_metadata.ipynb` for column inventory and schema checks
- `notebooks/clinvar/03_audits.ipynb` for split sanity checks, label balance, and failure cases

The important part is not the exact filenames. It is that each notebook answers one question and that
the final conclusion is mirrored in a short markdown note like this one.

## Open Consistency Check

One thing to keep in mind while consolidating the notebook trail: the current cache builder preserves
sequence-level windows and variant offsets, while the dataset tests also assert round-tripping of the
explicit allele strings. If the notebooks rely on explicit allele columns, make sure the final source
of truth is the parquet schema rather than an ad hoc notebook dataframe.

## Maintenance Rule

Whenever the curation logic changes, update:

- this page
- `docs/eval/clinvar.md`
- any notebook that encodes the same decision

That keeps the evaluation docs and the curation trail aligned.
